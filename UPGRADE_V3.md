## 1. 升级背景

### 用户原始需求（逐条记录）

> "所以为什么会没有景点刚才，这个点能彻底修复吗"
> "不希望你使用固定景点哈，我希望是实时的"
> "你列清单，对目前的功能进行检查，有没有明显的bug和不合理之处进行修复"
> "我希望景点之间的连线可以做成交通线路"
> "demo还差哪些"
> "继续进一步完成"
> "多方案对比可以放在开始规划之后吗，让用户选，而且用户可以同步对比三个方案，三个方案在用户打开过之后，其他两个并不会消失，而是可以返回继续查看，或者在网页端直接三个方案一并展示，分为三列，让用户同步观看"
> "用户既然选好预算，我认为就要在这个预算中分为三类，帮助用户在价格预算内节省，让用户有选择，是追求性价比、轻松游，还是深度体验。我的理念就是让用户在价格预算里有满足的体验"
> "旅行偏好的标签需要保留，在可以单选或者多选的情况下，不过可以加一个输入框，让用户添加其他信息，这个是选填的"
> "不需要加多方案对比这个按键，而是在开始规划后我们自动就开始多方案设计"
> "生成规划时的进度条展示要对应"
> "少了最后测试调适验收阶段，这一步很关键，我只要完整的成品"
> "我希望你编写一个升级文档，留存我给你的提示词，和你的计划"

### 需求分析

| 需求点 | 当前状态 | 目标状态 |
|--------|---------|---------|
| 多方案对比 | 有独立按钮，3个方案按预算分档 | 删掉按钮，开始规划自动出3个方案，按旅行风格分 |
| 预算维度 | 经济/舒适/豪华三档 | 统一预算内，三种体验策略 |
| 偏好选择 | 6个radio单选 | 多选标签 + 选填文本框 |
| 方案命名 | 固定"经济方案/舒适方案/豪华方案" | AI根据城市+偏好动态生成 |
| 对比体验 | 点选一个后另外两个消失 | 三列并排同步展示，可独立选择 |
| 进度条 | 4步，与实际流程脱节 | 5步，实时反映三个方案生成进度 |
| 移动端 | 未适配 | 左右滑动卡片 + 标签栏切换 |
| 验收测试 | 无 | 端到端完整测试 |

- `TripPlanRequest` 偏好字段：`preferences: str` → `preferences: list[str]`
- 新增 `extra_info: str` 字段（选填）
- 前端提交时合并多选标签 + 自由文本

#### 第2轮：后端数据模型适配（后端架构师）

**改动范围：** `schemas.py`, `routes.py`

**具体改动：**
- `TripPlanRequest` 模型改造：偏好从字符串改为列表 + 补充信息
- `/plan` 单方案接口暂时保留（用于降级）
- `/plan/variants` 接口重写：接收新格式偏好，按三方向生成
- 后端根据多选偏好加权：选了"美食"的用户，三个方案都侧重美食，但方式不同

#### 第3轮：Prompt三方向设计（AI prompt工程师）

**改动范围：** `prompts.py`

**具体改动：**

为三个方向各设计独立的 prompt 策略：

| 方案 | Prompt策略 | 景点选择 | 美食安排 | 交通方式 | 节奏 |
|------|-----------|---------|---------|---------|------|
| 经典高效 | 覆盖TOP景点，路线不走回头路 | 知名景点为主 | 标准3餐 | 公共交通 | 紧凑 |
| 轻松休闲 | 精选景点，留足休息和自由活动时间 | 精选2-3个 | 景点附近便捷 | 打车为主 | 舒缓 |
| 深度探索 | 小众景点+在地文化+特色美食 | 小众+知名混合 | 深挖本地美食 | 混合交通 | 适中 |

- 方案名由AI根据城市+偏好动态生成，不固定模板
- 三个方案允许20-30%的景点差异（如经典去西湖，深度可能去九溪烟树）
- 预算在同一区间内浮动（如"中等"：经典¥500/天、轻松¥600/天、深度¥700/天）

#### 第4轮：并行生成 + 流式进度（后端架构师）

**改动范围：** `routes.py`

**具体改动：**
- 三个方案 `ThreadPoolExecutor(max_workers=3)` 并行生成
- SSE 流式推送每个方案的生成状态（方案1完成/方案2生成中/...）
- 前端进度条实时反映三个方案的生成进度
- 进度条5步：景点搜索→天气查询→住宿推荐→三方案设计→汇总整理

## 3. 关键设计决策

### 3.1 预算体系

| 档位 | 每日预算区间 | 三方案浮动示例 |
|------|------------|--------------|
| 经济型 | ¥200-400/天 | 经典¥280 / 轻松¥320 / 深度¥380 |
| 中等 | ¥400-800/天 | 经典¥500 / 轻松¥600 / 深度¥700 |
| 豪华 | ¥800+/天 | 经典¥900 / 轻松¥1000 / 深度¥1200 |

三个方案都在用户选的预算区间内，但分配策略不同：
- 经典：景点多→门票占比高，餐饮住宿适中
- 轻松：景点少→门票占比低，住宿餐饮占比高
- 深度：小众景点门票低，但特色美食+体验项目占比高

### 3.2 首页改动

- 删掉"多方案对比"按钮，"开始规划"直接触发三方案流程
- 偏好：6个radio → 多选标签（至少选1个）+ 选填文本框
- 默认天数保持3天不动

### 3.3 进度条5步

```
Step 1: 🔍 景点专家搜索中...          (0-15%)
Step 2: 🌤️ 天气专家查询中...          (15-30%)
Step 3: 🏨 住宿专家推荐中...          (30-40%)
Step 4: 📋 规划专家设计三个方案...      (40-90%)
         ├─ 经典方案 ✅  轻松方案 ⏳  深度方案 ⏳
Step 5: 📝 整理专家汇总中...          (90-100%)
```

### 3.4 方案命名规则

AI 根据城市+偏好动态生成，示例：

| 城市 | 偏好 | 经典方案名 | 轻松方案名 | 深度方案名 |
|------|------|-----------|-----------|-----------|
| 杭州 | 美食+文化 | 西湖经典全览 | 杭州悠然时光 | 杭城寻味之旅 |
| 成都 | 美食 | 蓉城必打卡 | 巴适慢生活 | 成都味道之旅 |
| 北京 | 历史 | 皇城经典巡礼 | 京华慢时光 | 北京深度探秘 |
| 西安 | 文化历史 | 长安经典之旅 | 古都悠然行 | 西安文化深度游 |

### 3.5 移动端方案

采用**全屏卡片 + 左右滑动**方案：
- 每次展示一个方案的完整卡片
- 左右 swipe 或点击顶部标签栏切换
- 底部指示器圆点显示当前位置
- 固定"选择此方案"按钮

选择理由：旅行比价场景下，左右滑动对比更符合直觉（类似电商商品对比）。

## 5. 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 三个方案LLM生成耗时过长 | 用户等待超过30秒 | 并行生成 + 流式进度反馈 |
| 三个方案内容高度相似 | 失去对比价值 | prompt中强制要求20-30%景点差异 |
| 移动端滑动体验差 | 用户流失 | 原生touch事件 + CSS transform |
| 一个方案失败影响整体 | 用户体验差 | 独立错误处理 + 降级到单方案 |
| 预算预估偏差大 | 用户信任度下降 | 明确标注"预估"字样 + 区间展示 |
